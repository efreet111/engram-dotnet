# Proposal: Memory Level 2 — .md Artifact Promotion

## Intent

Las observaciones en engram-dotnet viven en una base de datos SQLite/Postgres. Son buscables por FTS5 pero no son versionables, revisables por humanos ni enlazables desde el código. Esto limita su valor como "memoria organizacional" — las decisiones importantes quedan atrapadas en una DB que nadie lee fuera del agente.

Esta propuesta introduce un **segundo nivel de memoria**: observaciones promovidas a archivos .md estructurados dentro del repositorio del proyecto, con link bidireccional (observation ↔ .md) y un template canónico.

## Scope

### In Scope
- Campo `md_path` en Observation (link observation → .md)
- Template engine para generar .md con frontmatter canónico
- Tool MCP `mem_promote_to_md` (observation_id → .md file)
- Tool MCP `mem_sync_md_to_repo` (batch: todas las observaciones sin .md → .md)
- Link bidireccional: observation.md_path + .md frontmatter observation_id
- Índice autogenerado `docs/decisions/index.md`
- Directorio destino configurable via `ENGRAM_MD_DIR` (default: `docs/decisions/`)

### Out of Scope
- Edición humana de .md con sync reverso a DB (fase posterior)
- Obsidian vault integration (ya existe en Fase B del roadmap)
- Templates personalizados por proyecto (se usa template fijo v1)

## Capabilities

### New Capabilities
- `md-promotion`: Capacidad de promover observaciones de la DB a archivos .md versionables en el repo. Incluye link bidireccional, template canónico, y herramientas MCP para promoción individual y batch.

### Modified Capabilities
- None (no cambia comportamiento existente de memoria)

## Approach

1. Agregar columna `md_path TEXT` a `observations` (SQLite) y campo equivalente en Postgres
2. Extender `Observation` model con `MdPath` + `AddObservationParams`/`UpdateObservationParams`
3. Implementar `MdTemplateEngine` — clase que recibe una Observation y genera .md con formato ADR
4. Implementar `mem_promote_to_md` — tool MCP que:
   a. Busca observation por ID
   b. Genera .md con frontmatter (observation_id, type, title, created_at, topic_key)
   c. Escribe .md en `{ENGRAM_MD_DIR}/{YYYY-MM-DD}-{slug}.md`
   d. Actualiza observation.md_path con la ruta del archivo
5. Implementar `mem_sync_md_to_repo` — tool MCP batch que escanea observaciones sin md_path y las promueve
6. Implementar `GenerateIndexAsync` — genera/actualiza `docs/decisions/index.md`

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| .md generados out-of-sync con DB si se editan manualmente | Med | Frontmatter `generated_at` + advertencia en template: "DO NOT EDIT — generated by engram" |
| Paths relativos vs absolutos en md_path | Baja | Usar paths relativos al repo root, configurables via ENGRAM_MD_DIR |
| Colisión de slugs en .md files | Baja | Incluir timestamp ISO en filename + hash corto si hay colisión |

## Rollback Plan

- `mem_promote_to_md` es aditivo — nunca modifica observaciones existentes (solo agrega md_path)
- Revocar: `UPDATE observations SET md_path = NULL WHERE md_path = '{path}'` + borrar .md
- `mem_sync_md_to_repo` tiene flag `--dry-run` para previsualizar

## Dependencies

- Ninguna — todas las columnas necesarias (`expires_at`, `review_after`) ya existen en el schema

## Success Criteria

- [ ] `mem_promote_to_md` genera .md con frontmatter correcto (observation_id, title, type, dates)
- [ ] Link bidireccional funciona: observation.md_path → file exists, .md frontmatter → observation existe
- [ ] `mem_sync_md_to_repo --dry-run` reporta N observaciones sin causar efectos secundarios
- [ ] `docs/decisions/index.md` se genera/actualiza correctamente
- [ ] Funciona en SQLite y PostgreSQL
- [ ] 280 tests existentes siguen pasando (sin regresiones)
